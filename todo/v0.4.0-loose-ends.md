# v0.4.0 loose ends

Loose ends identified during the v0.4.0 redesign session. Grouped by urgency: actionable now, blocked on browsers, and one large future project.

## Actionable now

### Light-theme accent-hi collision

After the AA palette fix darkened `--accent` from `#2d7ff9` to `#2d6cf4`, the light theme's `--accent-hi` (`#1f6ce0`) is nearly indistinguishable from `--accent` (contrast ratio 1.03:1 between them). Adjust `--accent-hi` in the light theme block of `tokens.css` to restore visible differentiation. The contrast test in `tests/test_contrast.py` will verify the new value still passes AA.

Affected files: `assets/css/tokens.css`
Effort: 5 minutes

### Visual review of palette and new primitives

The v0.4.0 palette changes and new primitives have never been seen by a human. Changes that need visual sign-off:

- `--text-faint` brighter in both themes (dark: `#64646c` -> `#8a8a94`, light: `#8d93a0` -> `#636b7a`)
- `--accent` darker (`#2d7ff9` -> `#2d6cf4`)
- `--border-2` significantly brighter (dark: `#2b2b2b` -> `#646464`, light: `#cfd3db` -> `#8e9298`)
- New primitives: checkbox, radio, file input, date picker, hovercard, tabs, segmented (with hidden radios)
- Responsive: drawer nav at 768px, touch targets at 44px, toast full-width on narrow screens
- RTL: sidebar, toast, switch in dir=rtl

Serve the gallery (`python3 -m http.server` at repo root, open `/gallery/`) and walk all routes in both themes at desktop, tablet (768px), and phone (360px) widths.

Effort: 30 minutes of looking

### E2e test flakiness

Two Playwright tests showed intermittent failures during the session:

- `forms.spec.js:37` ("form submission includes checkbox and radio values") — passes on re-run
- `tooltip-hovercard.spec.js` — fixed from `toBeVisible` to `toBeAttached` but the underlying timing sensitivity may recur

Investigate with more generous `waitFor` patterns or explicit state assertions before the interaction step.

Affected files: `tests/js/e2e/forms.spec.js`, `tests/js/e2e/tooltip-hovercard.spec.js`
Effort: 1 hour

### Number input spinner buttons

`input[type="number"]` is styled (Phase 11 fix) but the native spinner arrows were hidden by `appearance: none` without a custom stepper replacement. Keyboard increment/decrement still works; the visual up/down affordance is missing. Add custom +/- buttons or styled pseudo-element arrows.

Affected files: `assets/css/primitives.css`, possibly `assets/js/controls.js` if JS buttons are needed
Effort: 1 hour

### Sidebar drawer polish

The hamburger toggle works but the drawer UX is minimal:

- No backdrop overlay behind the open drawer (tapping outside closes it, but there's no visual dimming)
- No slide animation on open/close
- No swipe-to-close gesture

All optional UX refinements, not accessibility gaps.

Affected files: `assets/css/shell.css`, `assets/js/shell.js`
Effort: half a day

### copyButton and kebabButton convention

These two primitives were reclassified as NOT_COMPONENTS (simple element factories) rather than migrated to the `createX(opts) -> {el, methods, destroy()}` convention. They return raw button elements, not `{el, ...}` instances. Architecturally defensible (they're stateless one-shot factories), but they're the only two primitives not following the unified shape.

Decision needed: leave as utilities, or wrap them for consistency.

Affected files: `assets/js/controls.js`, `assets/js/index.js`, `tests/js/unit/api-convention.test.js`
Effort: small if wrapping, zero if leaving

## Blocked on browsers

### Customizable select (appearance: base-select)

Standing commitment: when `appearance: base-select` reaches Baseline (all browsers), migrate from the hybrid custom-menu + hidden-native-select to the fully styleable native select and delete the custom menu code. As of mid-2026, Chrome-only stable; Safari in Technology Preview; Firefox prototyping.

Track: caniuse customizable-select, web-features-explorer
Affected files: `assets/js/select.js`, `assets/css/primitives.css`

### Dialog closedby attribute

The modal has a JS light-dismiss shim (pointerdown on dialog element checks if click was on backdrop area) because Safari lacks the `closedby` attribute. When Safari ships it, the shim can be removed and `closedby="any"` used directly on the dialog element.

Track: caniuse dialog-closedby, WebKit release notes
Affected files: `assets/js/modal.js`

### popover=hint for tooltips

Tooltips currently use z-index positioning, not the Popover API. `popover="hint"` would give them proper top-layer stacking without closing auto-popovers (the exact tooltip use case). Blocked by Safari (Chrome and Firefox ship it).

Track: caniuse popover-hint
Affected files: `assets/js/tooltip.js`, `assets/css/primitives.css`

### CSS anchor positioning

The kernel's JS-based `placeBelow` was deliberately structured for future replacement by CSS anchor positioning (`anchor-name`, `position-anchor`, `position-area`). Baseline since January 2026 but only ~6 months old — the browser tail (users on older versions) makes it risky for a no-fallback framework. When the tail is comfortable (mid-2027+), the JS positioning module can be replaced with pure CSS.

Track: caniuse css-anchor-positioning
Affected files: `assets/js/kernel.js`, `assets/css/primitives.css`

## Large future project

### Tree-sitter checker rewrite

The v0.4.0 checker improvements were done via regex — 6 bypass classes closed, 3 FP classes fixed. But the regex architecture has inherent limits. Known remaining bypasses:

- HTML-in-JS-strings (`innerHTML = '<select>...'`) — the biggest gap; a real JS parser would see string content
- CSS `var()` resolution — `border-radius: var(--r)` always fires even if the token is 0; a real CSS parser with token-value tracking could resolve this
- JS string concatenation — `"https://" + host` is invisible to regex
- eval/Function — dynamically constructed code

The groundwork is in place: `py-tree-sitter` 0.26, `tree-sitter-javascript` 0.25, and `tree-sitter-css` 0.25 all have prebuilt PyPI wheels. `tinycss2` 1.5 is available for CSS tokenization. The checker's test fixtures already cover the bypass scenarios (they'll flip from clean to caught as the parser improves).

Architecture: replace the three scan functions (`scan_css`, `scan_js`, `scan_html` embedded delegation) with AST-query-based visitors. Keep the rule IDs and violation model unchanged so the CLI and test infrastructure don't change.

Affected files: `tinymoon/checker.py`, `pyproject.toml` (new deps: tree-sitter, tree-sitter-javascript, tree-sitter-css, tinycss2)
Effort: multi-session project (the regex scanner is 800 lines; the AST replacement will be comparable)
